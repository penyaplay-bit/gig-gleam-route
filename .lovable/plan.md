# Artist Credibility Profile

Turn `artist_owner_profiles` / `artist_rosters` into a promoter-facing credibility surface. The hero deliverable is a single **Booking Intelligence Score (0–100)** composed of 5 sub-scores, backed by six profile sections and a per-gig **Compatibility Score**.

## Scope

- Public route: `/artists/$slug` — the credibility view promoters land on.
- Owner route: `/_signedin/artist/profile` — the artist edits everything here.
- Compatibility widget on `/find-gigs/$id` (promoter view) and on the manager application form.
- Booking Intelligence Score computed server-side, cached on the profile row, recomputed nightly + on profile edit.

## The six sections (profile page)

1. **Reach (Audience)** — Instagram, TikTok, Facebook, YouTube, X, Spotify monthly listeners, Apple Music, Audiomack, Deezer. Total social reach ticker up top.
2. **Performance History** — total shows, shows this year, festivals, international, avg attendance, largest crowd, repeat-booking rate, years active.
3. **Reputation (Trust Score)** — 0–100 badge, star rating, breakdown chips: on-time, cancellation rate, promoter reviews, audience feedback, disputes, response speed, contract completion, profile completeness.
4. **Awards & Recognition** — badge grid (SAMA, Metro FM, AFRIMA, MTV, Apple Editorial, Spotify Viral, Platinum/Gold, Verified Artist, etc.).
5. **Commercial Value** — avg ticket sales, estimated pull per venue class (Small / Medium / Festival / Corporate), booking demand, avg fee band, merch performance, streaming growth, engagement rate.
6. **Media Kit** — bio, photos, logo, stage plot, tech rider, hospitality rider, videos, music links, press kit, one-click downloads.

Plus three sidecar modules:
- **AI Insights** — best-suited-for, audience demographics, peak days, estimated impressions, momentum %, regional strength. Generated by Lovable AI from the raw stats above.
- **Availability** — Available / Busy / On Tour / Booked / Tentative + month calendar.
- **Verified Identity** — checklist badges (Government ID, Phone, Email, Management, Bank, Tax).

## Booking Intelligence Score

Single 0–100 number = weighted average of five sub-scores:

| Sub-score      | Weight | Signals |
|----------------|--------|---------|
| Reach          | 25%    | Social + streaming totals, normalised by genre band |
| Reliability    | 25%    | Attendance %, cancellations, punctuality, disputes |
| Performance    | 20%    | Reviews, repeat bookings, event history depth |
| Momentum       | 15%    | 90-day streaming Δ, follower Δ, application → booking rate |
| Business       | 15%    | Response time, contract completion, on-time payments |

Displayed as: big number + tier label (Rising / Established / Elite / Icon) + radar chart of the 5 sub-scores.

## Per-gig Compatibility Score

On any gig detail, compute `matchScore(artist, gig)` returning 0–100 + 5 boolean checks: Genre, Budget, Audience, Location (distance/routing), Availability. Shown as `93% Perfect Match` with green/amber chips.

## Data model (new)

```text
artist_stats                    (1 row per artist, refreshed nightly)
  artist_id (FK), reach_json, history_json, momentum_json,
  intelligence_score int, tier text, subscores_json,
  updated_at

artist_socials                  (per platform, user-editable)
  artist_id, platform, handle, followers int, verified bool, updated_at

artist_awards                   (badge list)
  id, artist_id, name, org, year, tier (platinum/gold/nominee/win)

artist_reviews                  (promoter → artist, post-gig)
  id, gig_id, promoter_id, artist_id, rating int, on_time bool,
  professionalism int, would_rebook bool, comment text, created_at

artist_media                    (media kit assets)
  id, artist_id, kind (photo/video/stage_plot/tech_rider/hospitality/press),
  url, caption, sort_order

artist_verifications            (identity checks)
  artist_id, kind (id/phone/email/management/bank/tax), status, verified_at
```

Extend `artist_owner_profiles` with: `bio text`, `demographics_json`, `regional_strength text[]`, `venue_fit_json`, `availability_status text`. Booking history sub-scores derive from existing `bookings` / `gig_applications` — no new writes there.

Every new public table gets GRANTs + RLS: read-open on `artist_stats/socials/awards/media` (public profile data), owner-write via `has_role('artist')` + `user_id = auth.uid()`. `artist_reviews` write-only by the booking promoter, read-open.

## Build order (5 slices)

1. **Schema + owner edit UI** — migrations, `/_signedin/artist/profile` with tabs for the six sections. Artist can fill everything; nothing is scored yet.
2. **Public credibility page** — `/artists/$slug` renders the six sections + AI Insights placeholder + availability + verification badges. SEO head with artist name/genre; og:image = hero photo from media kit.
3. **Scoring engine** — `src/lib/artists/scoring.server.ts` computes 5 sub-scores + Intelligence Score. Nightly cron via `/api/public/cron/artist-scores` recomputes for all active artists; also triggered on profile edit.
4. **AI Insights** — server function calls Lovable AI (`google/gemini-3-flash-preview`) with the artist's stats blob, returns structured insights (Zod-validated), cached on `artist_stats.momentum_json`.
5. **Compatibility Score** — pure function `matchScore(artist, gig)` used on `/find-gigs/$id` header and the manager application form. No DB writes.

## Out of scope (this plan)

- Automated social-graph scraping (followers entered manually in v1; OAuth ingestion later).
- Payments/merch analytics beyond what the platform already tracks.
- Tour-routing optimizer.
